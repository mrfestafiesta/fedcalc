# Dynamic Tracker: Decoupled AR & Routing App

=========================================
OVERVIEW
=========================================
A Progressive Web App (PWA) designed for offline, multi-modal wilderness navigation. The application provides offline 3D hiking routes, live AR (Augmented Reality) trail breadcrumbs, and U.S. Federal Land overlays. It utilizes a custom routing engine to gracefully hand off navigation from global street API mapping to private, custom-built topology graphs.

=========================================
CORE FILE ARCHITECTURE
=========================================
1. index.html (The Shell App)
The primary interface handling UI rendering, Map logic, GPS tracking, Dijkstra routing math, and Augmented Reality viewport calculations. 
Key DOM Structures:
- #camera-container: Absolute-positioned viewport that hosts the live video feed. AR breadcrumbs and bubbles are injected here via JS.
- #controls: A flex grid stack utilizing max-height and overflow-y constraints to ensure core buttons (routing, camera toggle, trip recording) remain safely accessible and scrollable on small mobile screens.
- #settings-modal: The management hub. Uses a CSS grid to present real-time byte calculations of the browser's Cache API (Map Tiles, App Data) and IndexedDB (GPS paths), providing granular manual cache clearing buttons and screen Wake Lock controls.

2. sw.js (The Service Worker / Vault)
Acts as the offline proxy. It intercepts all network requests and routes them to four isolated cache buckets:
- tracker-shell-v1: HTML, CSS, JS.
- map-tiles: Aggressively caches OSM street tiles, Esri satellite imagery, and BLM federal boundaries.
- tracker-data-v1: Caches the `parks` directory and JSON topology graphs.
- weather-data-v1: Uses a "Network-First" strategy to fetch live forecasts, falling back to cached versions if cell service drops.

3. manifest.json
The Web App Manifest. Defines the app icon, theme color (#28a745), and allows the browser to install the app natively on Android/iOS homescreens.

4. parks/loc_manifest.json & parks/[id].json
The database structure. loc_manifest.json acts as a simple array index to populate the UI dropdown menu. The individual park files (e.g., lmuTest.json) contain the structured node dictionary, edge dictionary, and POI arrays generated by the Map Editor tool.

=========================================
CORE JAVASCRIPT FUNCTIONS (index.html)
=========================================
- loadParkMenu(): Fetches loc_manifest.json and dynamically populates the main park-selector <select> dropdown.
- clearARWorld(): Purges all dynamically spawned HTML tracking nodes from the video viewport to reset the AR scene.
- buildARWorld(): Determines which park topology nodes (breadcrumbs and POIs) exist within 300 feet of the user and spawns AR HTML elements for them.
- spawnARNode(targetData, type, content): Factory function that injects a stylized DOM element into the #camera-container.
- updateARPositions(): The AR Physics Engine. Compares target coordinates against the active hardware compass bearing and moves DOM elements along the X-axis of the screen using spherical trigonometry to mimic a HUD.
- toggleCompassUI(turningOn): Helper function that transitions the button styling and debug console visibility based on compass state.
- handleOrientation(e): Interprets the hardware 'deviceorientation' payload, neutralizing differences between iOS webkit sensors and raw Android Euler angles to establish a 0-360 degree compass bearing.
- getClosestNode(lat, lng, graphData): A snap-to-grid utility. Scans the active offline topology dictionary to find the nearest physical mathematical node to the user's GPS position.
- findShortestPath(startNodeId, endNodeId, graphData): Core Offline Math Engine. Executes Dijkstra's Algorithm using 3D Pythagorean weighting (distance + elevation changes). Imposes a heavy mathematical discount on 'drive' edges to bias vehicle routing over foot travel. Returns a path array and the total mathematical cost.
- formatRouteDist(feet): Utility to clean up string rendering in the UI (e.g. converting "8000 ft" into "1.5 mi").
- requestWakeLock(): Hooks into the navigator.wakeLock API to prevent the user's mobile display from dimming or locking while hiking.
- loadSavedTrip(): IndexedDB bootloader. Fires on initialization to retrieve and re-paint any recorded GPS paths from a previous/interrupted session.
- getCacheSizeMB(cacheName, isTiles): Inspects Service Worker storage vaults to calculate active byte weight (or tile approximations) for the settings UI.
- getPathSizeMB(): Translates the raw IndexedDB coordinate array into a virtual blob to establish a measurable byte footprint for the settings UI.
- refreshStorageNumbers(): Invokes all size calculators and updates the settings grid DOM elements.
- showStatusPill(message, bgColor): Fires a transient, CSS-animated notification banner at the top of the screen.

=========================================
EXTERNAL APIS & DATA QUERIES
=========================================
Our architecture relies heavily on third-party integrations for live data context. Below is the structure of our dependencies:

1. OpenStreetMap (Base Layer)
- URL: https://{s}.tile.openstreetmap.org/{z}/{x}/{y}.png
- Return: 256x256 pixel PNG tiles.
- Structure: Standard Z/X/Y geospatial URL querying. Completely open-source, requires no authentication, and serves as our default street and boundary visualizer.

2. Esri / ArcGIS (Satellite Layer)
- URL: https://server.arcgisonline.com/ArcGIS/rest/services/World_Imagery/MapServer/tile/{z}/{y}/{x}
- Return: 256x256 pixel PNG tiles.
- Structure: High-fidelity raster imagery used for visual terrain orientation.

3. Bureau of Land Management (Federal Overlay Layer)
- URL: https://gis.blm.gov/arcgis/rest/services/lands/BLM_Natl_SMA_Cached_without_PriUnk/MapServer/tile/{z}/{y}/{x}
- Return: 256x256 pixel PNG tiles (or HTTP 404).
- Structure: Publicly hosted ArcGIS MapServer. 
- Quirks/Dependencies: To save server space, the US Government does not cache transparent tiles over oceans or massive urban centers. If queried for an un-mapped coordinate, it forcibly drops the connection with an HTTP 404. We mitigate browser overload by applying a geographic bounding box directly to the Leaflet instantiation (`bounds: [[14.0, -180.0], [72.0, -60.0]]`) to cull off-continent requests.

4. Open-Meteo (Weather & Global Elevation)
- URL: https://api.open-meteo.com/v1/forecast?latitude=X&longitude=Y&current_weather=true
- Return: JSON payload.
- Structure: An open-source, free-use meteorological database. We use this endpoint for two distinct functions. In the Map Editor, we use it for bulk querying terrain elevation points worldwide. In the Main App, we tap the `current_weather` object to inject live temperatures and windspeeds into POI popups. 

5. USGS National Map (High-Fidelity Elevation)
- URL: https://epqs.nationalmap.gov/v1/json?x=LONG&y=LAT&units=Meters
- Return: JSON payload (`{"value": 150.5}`).
- Structure: The gold standard for US-bound elevation data. 
- Quirks/Dependencies: The USGS servers are notoriously fragile and aggressively rate-limited to protect public infrastructure. In our Map Editor tool, we strictly batched our HTTP fetch requests (Chunking 10 at a time with 300ms delays) to prevent IP bans or HTTP 503 Service Unavailable timeouts. In the Main App, we use this endpoint conservatively on single POI clicks to overwrite Open-Meteo data with a higher degree of topographical accuracy.

6. Project OSRM (Global Street Routing)
- URL: https://router.project-osrm.org/route/v1/driving/START_LNG,START_LAT;END_LNG,END_LAT?geometries=geojson
- Return: JSON payload containing a GeoJSON linestring array and mathematical distance.
- Structure: An open-source routing engine built on OpenStreetMap data. 
- Implementation Logic: Because our custom park networks do not know how to navigate users down public highways, our engine executes a "Handoff." If a user is far from the trail, we pass their GPS coordinate and the coordinate of the nearest park `Gate Node` to OSRM. OSRM returns a GeoJSON array, which we flip (OSRM returns Lng/Lat, Leaflet expects Lat/Lng) and render as the "Blue Line", seamlessly terminating exactly where our internal Dijkstra algorithm picks up the "Brown Line" offroad logic.
